# V2 Plan

## Goal

Evolve the current Tilescript layout model instead of replacing it.

Keep:

- `workspace`
- `group`
- `slot`
- `window`
- JSX authoring
- CSS styling

Improve:

- movement semantics
- resize semantics
- authored region semantics
- nested boundary behavior

This is a semantic evolution, not a syntax rewrite.

## Current Realization

The current API is already close to the right shape.

For example:

```tsx
<workspace id="frame">
  <window
    id="nitrogen-column"
    match='class="Nitrogen"'
    class="nitrogen-column"
  />

  <group id="main-area">
    <slot id="master" take={1} class="master-slot" />

    {mainWindowCount > 1 ? (
      <group class="stack-group">
        <slot id="stack-slot" class="stack-group__item" />
      </group>
    ) : null}
  </group>
</workspace>
```

This already expresses:

- explicit authored structure
- nested regions
- singleton matched content
- CSS-owned styling

So V2 should answer behavioral questions around this structure rather than invent a separate API.

## Core Direction

### 1. Authored Structure Stays Primary

The authored JSX tree is the primary source of truth.

That means:

- `workspace` and `group` define structural boundaries
- `slot` and `window` define how windows are claimed into leaves
- runtime state may override sibling order within a structural boundary
- authored order remains the fallback when there is no runtime override

Important default:

- authored sibling order should feel fixed unless the author explicitly opts into movable-group semantics

This matches normal React expectations: if a child is written first in JSX, authors will usually expect it to stay first unless they asked for dynamic reordering.

### 2. Structure Alone Is Not Enough

Tree shape defines:

- movement scope
- sibling sets
- nesting boundaries

But tree shape alone does not determine all movement behavior.

Example:

```tsx
<workspace id="frame">
  <window id="nitrogen-column" match='class="Nitrogen"' />
  <group id="main-area">...</group>
</workspace>
```

and:

```tsx
<group id="main-area">
  <slot id="master" take={1} />
  <group class="stack-group">...</group>
</group>
```

are structurally analogous, but we do not want identical behavior:

- `master` should be able to move into `stack`
- `nitrogen-column` should not automatically be absorbed into `main-area`

So movement semantics need both:

- structure
- node role

### 3. Prefer Structural Semantics Over Leaf Special-Casing

The cleaner direction is:

- keep `window` close to `slot take={1}` semantically
- let explicit authored containers carry the important region behavior

Recommended refinement:

- all explicit `group`s are structural boundaries by default
- some `group`s may opt into moving as a unit for plain move commands
- use an explicit prop such as `moveAs="group"`

That means if an author wants singleton or multi-window region-as-a-unit behavior, they should author an explicit `group` and opt that `group` into unit movement.

Example:

```tsx
<workspace id="frame">
  <group id="nitrogen-region" moveAs="group">
    <window
      id="nitrogen-column"
      match='class="Nitrogen"'
      class="nitrogen-column"
    />
  </group>

  <group id="main-area">
    <slot id="master" take={1} class="master-slot" />

    {mainWindowCount > 1 ? (
      <group class="stack-group">
        <slot id="stack-slot" class="stack-group__item" />
      </group>
    ) : null}
  </group>
</workspace>
```

Under this interpretation:

- `nitrogen-region` and `main-area` are top-level sibling regions
- the Nitrogen window is leaf content inside `nitrogen-region`
- important movement semantics belong to the authored `group`, not the `window`
- plain move commands from focused descendants may move the whole group when `moveAs="group"` is set

## Movement Model

### 1. Two Separate Operations

The system should distinguish:

1. window movement
2. region movement

These are not the same and should not silently collapse into each other.

But for user-facing commands, plain move may resolve to either operation depending on authored semantics.

### 2. Window Movement Searches Outward

For window movement:

1. start at the focused window
2. walk outward through authored ancestors
3. find the nearest structural boundary where the requested transfer is valid
4. perform window membership transfer at that boundary

This preserves cases like:

- master window moving into stack
- window moving from one authored sibling group into another authored sibling group

Concrete example:

```tsx
<workspace>
  <group>
    <slot take={2} />
  </group>

  <group>
    <slot take={2} />
  </group>
</workspace>
```

If focus is on the first window in the first group and the user presses move-right:

- do not move the whole first group
- do not no-op just because the immediate local node does not provide a move
- walk outward to the nearest valid authored boundary
- move the focused window into the right group

### 3. Region Movement Is Separate, But Can Be Selected By Plain Move

Region movement should operate on explicit authored regions, not on inferred partitions.

For example, in:

```tsx
<workspace id="frame">
  <group id="nitrogen-region" moveAs="group">
    <window id="nitrogen-column" match='class="Nitrogen"' />
  </group>

  <group id="main-area">...</group>
</workspace>
```

region movement at the `frame` boundary can reorder:

- `nitrogen-region`
- `main-area`

But normal window movement inside `main-area` should remain local to `main-area`.

Recommended rule:

- plain move first tries local window transfer
- if the focused window is inside an ancestor `group` with `moveAs="group"`, plain move may resolve as movement of that group among sibling regions at the nearest valid boundary
- there should be no surprising escalation to whole-region movement unless the author explicitly opted the group into that behavior

### 4. Current Master/Stack Behavior Must Stay Valid

For a plain master-stack layout like:

```tsx
<workspace id="frame">
  <group id="main-area">
    <slot id="master" take={1} class="master-slot" />

    {mainWindowCount > 1 ? (
      <group class="stack-group">
        <slot id="stack-slot" class="stack-group__item" />
      </group>
    ) : null}
  </group>
</workspace>
```

the master window should still be able to move into the stack region.

That is a local `main-area` window movement operation.

### 5. Nitrogen-Like Singleton Region Behavior

For a Nitrogen-style singleton:

```tsx
<workspace id="frame">
  <group id="nitrogen-region" moveAs="group">
    <window
      id="nitrogen-column"
      match='class="Nitrogen"'
      class="nitrogen-column"
    />
  </group>

  <group id="main-area">...</group>
</workspace>
```

recommended behavior is:

- Nitrogen moves as part of `nitrogen-region`
- Nitrogen does not get absorbed into `main-area`
- `nitrogen-region` and `main-area` may be reordered as sibling regions at the `workspace` boundary

This is now the recommended UX behavior for plain move commands too, not just a separate region-move command.

## Boundary Ownership

Recommended ownership rules:

- `workspace` owns top-level sibling region order
- `group` owns local sibling region order
- leaf nodes do not own child order state

This keeps ordering attached to authored containers rather than inventing a separate pane tree.

## Remaining Design Decision

One important design question remains:

What is the final shape and naming of the explicit group-level move-unit prop?

The direction above suggests:

- `workspace` and `group` are the main candidates
- `window` and `slot` are leaf claim nodes

Current preferred answer:

- all explicit `group`s are structural regions by default
- `moveAs="group"` opts a `group` into moving as a unit for plain move commands
- without that opt-in, authored sibling order should remain fixed for that group

`regionMovable` was only a temporary idea and is not the current plan.

`moveAs="group"` is the current best candidate.

Example:

```tsx
<workspace id="frame">
  <group>
    <window
      id="nitrogen-column"
      match='class="Nitrogen"'
      class="nitrogen-column"
    />
  </group>

  <group id="main-area">...</group>
</workspace>
```

Recommended reading:

- the first group is authored first
- so it should stay first by default
- plain move on Nitrogen should not reorder that group unless the author explicitly opted in

If the author wants it movable as a unit, they should say so explicitly:

```tsx
<workspace id="frame">
  <group id="nitrogen-region" moveAs="group">
    <window
      id="nitrogen-column"
      match='class="Nitrogen"'
      class="nitrogen-column"
    />
  </group>

  <group id="main-area">...</group>
</workspace>
```

## Additional Questions That Arise From Flexible Authored Layouts

These should be answered before implementation:

### 1. What counts as a structural sibling set?

If a `group` contains a mix of:

- `window`
- `slot`
- nested `group`

then movement among those siblings must be clearly defined.

Recommended answer:

- the direct authored children under the same `workspace` or `group` form the sibling set

### 2. What happens when a matched region disappears?

Recommended answer:

- preserve order state by region id
- omit absent regions from the rendered sibling set
- when they reappear, merge them back according to remembered order

### 3. What happens when a region contains multiple windows?

Recommended answer:

- window move defaults to local movement first
- explicit `moveAs="group"` allows group-as-a-unit movement for plain move commands

### 4. How should recursive nesting behave?

Recommended answer:

- movement resolution walks outward from the focused node
- each authored parent boundary is tried in turn
- the first valid boundary wins

### 5. How should resize interact with authored regions?

Recommended answer:

- resize should follow the same authored-boundary model as move
- local sublayout resize stays local
- explicit sibling region resize applies at the authored container boundary

### 6. How should authored order and runtime order combine?

Recommended answer:

- authored order is the default
- runtime state overrides direct sibling order only
- if runtime state is missing or stale, authored order wins

## Current Recommendation

1. Keep the current syntax.
2. Make V2 a semantic evolution of the current model.
3. Keep movement centered on authored boundaries.
4. Distinguish clearly between window movement and region movement.
5. Prefer structural wrapper regions over special-casing leaf `window` semantics.
6. Make all explicit `group`s structural by default.
7. Use an explicit group-level opt-in such as `moveAs="group"` when plain move should move the whole group as a unit.
